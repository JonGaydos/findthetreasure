Design: Technical Architecture

Next.js full-stack — one Docker container, no database, stateless by design

How the Share Code Works

Share Code Lifecycle
User 1 clicks map
Sets tolerance + hint
→
POST /api/create-game
Server encrypts payload with
AES-256-GCM + app secret key
(env var in Docker)
→
FTT-X7K2-9MQP
↓ User 1 shares code via text/Discord/etc.
User 2 enters code
on their own Docker instance
→
POST /api/guess
Code + guess coords sent up.
Server decrypts code → computes
Haversine distance → returns
distance only
→
"0.6 miles away"
coords never leave server

API Routes (3 total)

POST /api/create-game Receives: lat, lng, tolerance, hint, hintAfterGuesses → Returns: share code
POST /api/guess Receives: shareCode, guessLat, guessLng → Returns: distance, isWin, isHintUnlocked, hint (if unlocked)
POST /api/reveal Receives: shareCode → Returns: treasure lat/lng (only called on win, loss, or give up)

State Management — No Database Needed

Server (stateless)
  • Encrypts/decrypts share codes
  • Computes distances
  • No session, no DB
  • Secret key in APP_SECRET env var
Browser (localStorage)
  • Share code (for re-sending with each guess)
  • All previous guesses + distances
  • Unit preference
  • Circle overlay toggle state
  • Hint unlocked flag

Docker Setup

# Single container, single port
FROM node:20-alpine
EXPOSE 3000
ENV APP_SECRET=change-this-secret
# Both users must use the same APP_SECRET to share codes

⚠️ Both User 1 and User 2 must run Docker with the same APP_SECRET — this is what allows the code to be decrypted. Share it once at setup time (like a household Wi-Fi password).
